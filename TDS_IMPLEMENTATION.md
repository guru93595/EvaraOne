# TDS (Total Dissolved Solids) Sensor System - Complete Implementation

## Overview
Complete end-to-end implementation of TDS water quality sensors in the EvaraTech platform. TDS sensors measure total dissolved solids in water (ppm = parts per million) and their quality levels.

## What Changed

### 1. Frontend Changes

#### a. Device Type Registration (AddDeviceForm.tsx)
- **Changed**: "Custom Sensor" → "TDS Sensor"
- **Icon**: Gauge (test tube icon for measurements)
- **Template**: Changed from "EvaraTank" reuse to new "EvaraTDS"
- **Description**: "Water quality TDS measurement sensors"

#### b. Database Types (database.types.ts, database.ts, entities.ts)
- Added "EvaraTDS" to `AnalyticsType` union type
- Added "EvaraTDS" to analytics_type enum
- Added "tds_sensor" to node_asset_type enum
- Added "TDSSensor" to node_category enum
- Updated DeviceCreate interface to support EvaraTDS analytics template

### 2. Backend Changes

#### a. Admin Controller (admin.controller.js)
- Added support for "evaratds" / "tds" / "EvaraTDS" device type validation
- Added TDS-specific metadata configuration:
  - Type: "TDS"
  - Unit: "ppm" (parts per million)
  - Min threshold: 0
  - Max threshold: 2000
  - Field mapping:
    - field1 → tds_value (TDS reading)
    - field2 → temperature (water temperature)

#### b. TDS Schema (tds.schema.js - NEW)
Validation schemas for:
- `createTDSDeviceSchema`: Provisioning new TDS sensors
- `updateTDSDeviceSchema`: Updating configuration
- `getTDSDeviceSchema`: Fetching device details
- `getTDSDataSchema`: Telemetry data validation

#### c. TDS Controller (tds.controller.js - NEW)
Complete API implementation with 5 endpoints:

1. **GET /api/v1/devices/tds/:id/telemetry**
   - Latest TDS reading
   - Water quality classification (Excellent/Good/Fair/Poor/Very Poor)
   - Device status (Online/Offline/Offline Recent)
   - Temperature reading
   - Last update timestamp

2. **GET /api/v1/devices/tds/:id/history?hours=24&limit=288**
   - Historical TDS readings
   - Timestamps for each reading
   - Quality classification timeline
   - Configurable time range

3. **GET /api/v1/devices/tds/:id/config**
   - Device configuration
   - Min/max thresholds
   - Location (latitude/longitude)
   - Sensor field mapping

4. **PUT /api/v1/devices/tds/:id/config**
   - Update thresholds
   - Update location
   - Caches invalidation

5. **GET /api/v1/devices/tds/:id/analytics?hours=24**
   - Average TDS value
   - Min/max TDS values
   - Average temperature
   - Reading count

#### d. TDS Routes (tds.routes.js - NEW)
- Authentication middleware applied to all routes
- RESTful endpoint structure
- Input validation using Zod schemas

#### e. Server Integration (server.js)
- Mounted TDS routes at `/api/v1/devices/tds`
- Applied global SaaS authentication
- Routes available for authenticated users

### 4. Data Flow

```
Frontend: Provision TDS Form
    ↓
Backend: Admin Controller (createNode)
    ↓
Add "evaratds" to devices collection registry
    ↓
Add metadata to evaratds collection
    ↓
Telemetry Worker (polls every 60s)
    ↓
Fetch from ThingSpeak using channel ID + API key
    ↓
Parse TDS value (field1) and Temperature (field2)
    ↓
Update device document with latest values
    ↓
Frontend: Call TDS API endpoints
    ↓
Get telemetry → Display on dashboard
```

### 5. Water Quality Classification

TDS levels (in ppm):
- **< 300 ppm**: EXCELLENT (pure water, distilled)
- **300-600 ppm**: GOOD (safe drinking water)
- **600-1000 ppm**: FAIR (acceptable, some treatment recommended)
- **1000-1500 ppm**: POOR (high mineral content, uncomfortable)
- **> 1500 ppm**: VERY_POOR (unsuitable for drinking)

### 6. ThingSpeak Integration

TDS sensors send data to ThingSpeak with:
- **Field 1**: TDS value (in ppm)
- **Field 2**: Water temperature (in °C)
- Timestamps automatically generated by ThingSpeak

Example ThingSpeak Response:
```json
{
  "field1": "425",          // TDS value
  "field2": "28.5",         // Temperature
  "created_at": "2026-04-15T13:07:26Z"
}
```

### 8. Caching Strategy

- TDS telemetry cached for 1 minute (`tds:telemetry:{id}`)
- Device list cache busted on updates (prefix `nodes_`)
- History data cached dynamically (based on request params)
- Analytics cached per device

### 9. Security & Access Control

- All TDS endpoints require authentication
- Ownership verification for non-superadmins
- Rate limiting applied via global middleware
- ThingSpeak API keys encrypted in Firestore
- Only API key hash stored for MQTT authentication

### 10. Error Handling

- 404: Device not found or wrong type
- 403: Unauthorized access (wrong customer/zone)
- 400: Missing ThingSpeak credentials or invalid parameters
- 500: ThingSpeak API failure or database error
- All errors logged with request ID for tracing
